Contexte, Audit & Architecture

Contexte & Objectifs

Audit, extension et documentation d'un prototype RAG existant pour en faire un assistant IA de production capable de répondre à des questions hybrides : textuelles (documents PDF) et statistiques (données Excel/SQL NBA).

🔒
Fiabilité
Passer d'un prototype à un outil robuste et traçable. Validation stricte des flux via Pydantic & monitoring en temps réel via Logfire.
🔀
Multimodalité
Combiner analyse textuelle (RAG sur documents PDF) et statistiques numériques (NL-to-SQL sur données Excel/SQLite NBA).
⚙️
Industrialisation
Validation stricte des flux de chunking/embedding + monitoring complet de la chaîne RAG/LLM pour un déploiement production.
📊
Livrable
Rapport complet de mise en place et d'évaluation comparative avant/après intégration SQL avec tableaux et graphiques RAGAS.

Audit du Prototype — Mesurer pour Améliorer

1
Framework RAGAS — Évaluation Objective
evaluate_ragas.py
Mesure automatisée de 3 métriques : Fidélité (la réponse est-elle fondée sur le contexte ?), Rappel du contexte (tout le contexte pertinent est-il récupéré ?) et Précision du contexte (le contexte récupéré est-il pertinent ?). 10 cas de test couvrant 3 catégories : SIMPLE, COMPLEX, NOISY.
2
Sécurisation des Flux — Validation Stricte
Pydantic & Pydantic AI
Validation typage du chunking et de l'embedding. Chaque chunk produit est validé avant injection dans le Vector Store. Les erreurs de format, de longueur ou de contenu sont détectées et loguées avant qu'elles n'atteignent le modèle.
3
Observabilité — Traçabilité Pas à Pas
Pydantic Logfire
Visualisation en temps réel de chaque étape de la chaîne RAG/LLM : réception question → routing → retrieval → génération → réponse. Identification des goulots d'étranglement et des cas d'échec.
4
Jeu de Tests Métier — 3 Catégories
10 cas de test · evaluate_ragas.py
3 cas SIMPLE : questions directes sur les statistiques NBA. 4 cas COMPLEX : questions hybrides texte + statistiques, comparaisons multi-joueurs. 3 cas NOISY : requêtes avec abréviations, fautes de frappe, langage informel (ex: "sga cmb de pts").

Architecture Fallback SQL ↔ RAG

Résolution automatique des requêtes ambiguës

Le système retrouve automatiquement le vrai nom d'un joueur si la requête SQL échoue — illustré par le cas réel : « sga cmb de pts ».

1
Question utilisateur : "sga cmb de pts" — abréviation informelle non reconnue par le SQL
Entrée
2
Agent SQL tente une requête sur la table players avec le terme "sga" → 0 résultats
❌ Échec SQL
3
Fallback RAG interroge le Vector Store FAISS → retrouve "SGA = Shai Gilgeous-Alexander" dans les documents
✅ RAG trouvé
4
Agent SQL relancé avec le vrai nom → 32.7 pts/match récupéré depuis season_stats
✅ SQL OK
5
Mistral API synthétise la réponse finale en combinant le contexte RAG et les données SQL
✅ Réponse

Intérêt du fallback : ce mécanisme transforme un échec SQL en opportunité d'enrichissement. Le RAG joue un rôle de résolution d'entités (NER implicite) qui complète le pipeline SQL — les deux routes sont complémentaires, pas concurrentes.

Intégration SQL & Résultats RAGAS

Intégration des Données Structurées — NL to SQL

🗃️ SQLite Schema
4 tables interconnectées : players · season_stats · stats · reports. Pipeline d'ingestion load_excel_to_db.py avec validation Pydantic à chaque étape d'import.
🤖 LangChain Tool
Outil sql_tool.py intégré dans l'orchestrateur LangChain. Génération dynamique SQL depuis la question en langage naturel → exécution → retour structuré vers Mistral.
📝 Few-Shot Optimization
Templates SQL améliorés par des exemples types (few-shot prompting). Réduit les erreurs de jointure et améliore la précision sur les requêtes statistiques complexes (tri, agrégation, filtres multi-critères).
🔍 Ingestion Pipeline
Validation Pydantic à chaque ligne importée depuis Excel. Détection automatique des doublons, des valeurs nulles et des types incohérents. Log structuré via Logfire.

Cas N01 — Régression analysée : le seul cas dégradé (−0.15) correspond à une question bruitée où le SQL retourne un résultat non pertinent qui pollue la synthèse Mistral. Le RAG seul, dans ce cas précis, produisait une réponse plus prudente. Piste : améliorer le filtrage de confiance avant injection SQL dans le contexte final.

Conclusions & Perspectives

Résultat principal : l'intégration SQL transforme radicalement les performances du système. Le score RAGAS passe de 0.292 à 0.800 (+174%), avec 9 cas sur 10 améliorés. Le prototype est devenu un agent IA de production fiable sur les questions statistiques NBA.

Limites identifiées

  • Mapping NL→SQL fragile sur les requêtes très ambiguës : le few-shot ne couvre pas tous les cas d'agrégation complexes (ex: classement multi-critères, fenêtres temporelles)
  • Biais de couverture : les données Excel importées couvrent une saison spécifique — questions sur d'autres saisons peuvent échouer sans fallback explicite
  • Cas NOISY N01 : injection SQL non filtrée qui dégrade la réponse dans certains cas bruités
  • Latence : le double appel (SQL + RAG + Mistral) augmente le temps de réponse pour les questions hybrides complexes

Prochaines étapes

  • 1Rédaction du rapport final d'évaluation avec tableaux et graphiques de performance RAGAS — livrable OpenClassrooms
  • 2Passage à l'échelle : adaptabilité du modèle pour d'autres clubs ou disciplines sportives (NFL, Ligue 1, F1…)
  • 3Filtre de confiance SQL : seuil de pertinence avant injection dans le contexte Mistral pour éviter les régressions NOISY
  • 4Validation production : tests de robustesse sur un panel utilisateur réel + validation de la latence acceptable pour déploiement final
  • 5Extension multimodale : intégration de données vidéo/image (highlights) pour enrichir les réponses narratives de Mistral
